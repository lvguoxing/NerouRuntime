# NeuroRuntime 企业级医疗软件开发计划

> 版本：v1  
> 目标：把 NeuroRuntime 收敛为“EEG/神经信号文件源训练 -> ONNX -> ONNX Runtime DATA -> 验证报告”的企业级桌面工作台。  
> 原则：少做无效功能，优先完成可交付闭环；所有高风险功能必须可追溯、可验证、可回滚。

## 1. 产品对象重新定义

NeuroRuntime 的当前核心对象不是“采集平台”，也不是“科研工具集合”，而是企业内部用于算法工程交付的 EEG/神经信号模型训练与推理包生成工作台。

核心对象：

| 对象 | 定义 | 交付要求 |
|---|---|---|
| Training Source | EEG/神经信号训练文件或标准训练目录 | 可预检、可解析、可记录来源 |
| Dataset Manifest | 数据集摘要、标签、通道、采样率、版本、hash | 训练前必须生成或校验 |
| Training Run | 一次训练任务及其参数、日志、指标、产物 | 必须可复现、可审计 |
| ONNX Model | 训练后导出的生产推理模型 | 必须带输入输出 contract |
| Runtime DATA Package | ONNX Runtime 生产推理包 | 必须包含模型、标签、预处理、通道 schema、样例输入输出、验证报告 |
| Validation Result | 离线或生产前推理验证结论 | 必须输出 deployable true/false 与原因 |
| Audit Event | 关键操作审计事件 | 必须记录操作者上下文、对象、时间、结果 |

当前 Runtime DATA 包除脚本生成的基础文件外，C++ 服务层还会补写：

| 文件 | 用途 |
|---|---|
| `runtime_contract.json` | 企业交付 contract，记录必需文件、deployable 状态、来源路径与检查结果 |
| `audit_events.jsonl` | 导出启动、拒绝、失败、完成等审计事件，一行一个 JSON |
| `manifest.json.enterpriseContract` | 将交付 contract 摘要回写到 manifest，方便生产侧快速判断 |

Dataset Manifest 已在代码中落地：

| 位置 | 责任 |
|---|---|
| `Domain::DatasetManifest` | 数据集 contract 对象，记录来源、形状、采样率、类别、文件清单和状态 |
| `MetadataSerializer::writeDatasetManifestJson` | 写出标准 `dataset_manifest.json` |
| `TrainPreflight::evaluate` | 自动识别 `dataset_manifest.json`，并阻止 C/T 形状不一致的数据进入训练 |
| `DatasetPreparationService` | 数据准备完成后写出 `dataset_manifest.json` |
| `TrainingService` | 训练启动前强制预检；缺少 manifest 时基于 NPZ 目录自动补齐 |

## 2. 角色与业务场景

| 角色 | 核心场景 | 禁止默认暴露的复杂度 |
|---|---|---|
| 算法工程师 | 选择数据、预检、训练、查看曲线、导出 ONNX | 不默认展示采集设备细节和长参数表 |
| 交付工程师 | 加载 ONNX、运行 Golden Sample、生成 Runtime DATA 包 | 不默认进入训练调参细节 |
| QA/验证工程师 | 比较模型版本、检查输入输出一致性、出具报告 | 不直接修改训练参数 |
| 项目负责人 | 查看最近产物、风险、是否可交付 | 不暴露底层日志，除非进入诊断 |
| 系统管理员 | 配置路径、加速模式、日志、后续权限 | 不参与业务页默认流程 |

MVP 不强制登录，但代码和数据结构必须为权限、审计、电子签名、版本冻结预留接口。

## 3. 业务流程

主流程只保留两页：

```text
模型训练 -> 推理模型
```

详细流程：

1. 选择训练文件源或训练目录。
2. 执行自动预检：路径、格式、NPZ 数量、通道数、采样率、标签、NaN/Inf、输入 shape。
3. 生成或读取 Dataset Manifest。
4. 使用默认训练配置启动训练；高级参数默认折叠。
5. 写入训练日志、指标、配置和 checkpoint。
6. 导出 ONNX，并写入模型 manifest。
7. 加载 ONNX 和 Golden Sample，执行 ONNX Runtime 验证。
8. 生成 Runtime DATA Package。
9. 生成验证报告与交付结论。

## 4. 信息架构与 UI 布局

### 4.1 顶层导航

当前主导航只允许：

| 导航 | 页面职责 |
|---|---|
| 模型训练 | 文件源选择、预检、训练运行、ONNX 产物 |
| 推理模型 | ONNX 加载、Golden Sample、Runtime DATA、验证报告 |

旧的总览、采集、预处理、归档页面保留代码兼容，不作为主导航或命令面板的默认入口。后续如需要，只能以专家模式、弹窗或侧路工具出现。

### 4.2 页面布局规范

每个页面采用三段式信息层级：

1. 顶部：当前对象、状态、主行动。
2. 主区：当前任务所需的最少控件和主要结果。
3. 底部或抽屉：日志、诊断、报告、专家参数。

页面必须回答五个问题：

1. 当前在哪一步？
2. 当前处理的对象是什么？
3. 当前对象状态是否可用？
4. 现在允许执行什么操作？
5. 执行后会生成什么产物？

## 5. 视觉与交互标准

| 类型 | 标准 |
|---|---|
| 主题 | 默认浅色，低疲劳，高信任，数据密集但不拥挤 |
| 色彩 | 中性色表面 + 蓝绿色主强调 + 橙/红风险语义 |
| 状态 | 不只靠颜色，必须有文字状态 |
| 控件 | 主行动每页最多一个；危险动作二级化并确认 |
| 长任务 | 必须有进度、日志、取消/停止、失败诊断 |
| 专家参数 | 默认折叠，保存最近使用配置 |
| 日志 | 业务页只显示摘要；完整日志进入系统日志面板 |
| 空状态 | 禁止空白容器，必须说明下一步 |

## 6. 医疗级软件开发要求

本项目现阶段定位为医疗/神经信号工程工具，不宣称临床诊断用途；但开发过程按医疗级软件工程纪律执行。

| 要求 | 落地方式 |
|---|---|
| 可追溯性 | 数据、配置、模型、报告都必须带版本、路径、hash、时间 |
| 风险控制 | 高风险动作前置预检，失败时阻止 deployable |
| 审计 | 训练、导出、验证、包生成写入审计事件 |
| 配置管理 | 默认配置、专家配置、运行配置分离保存 |
| 验证确认 | 单元测试、集成烟测、Golden Sample 回归 |
| 数据完整性 | 输入文件不可静默修改；派生产物写 manifest |
| 错误处理 | 错误信息必须可操作，禁止只写“失败” |
| 网络与隐私 | 默认本地运行；后续联网功能必须显式开关 |
| 构建发布 | Release 构建、依赖版本、ONNX Runtime EP 需记录 |

推荐参考流程：IEC 62304 的软件生命周期思想、ISO 14971 的风险管理思想、ISO 13485 的配置与记录思想。当前不声称已认证，但代码和文档应朝这些要求组织。

## 7. 系统架构要求

| 层 | 职责 |
|---|---|
| Domain | 数据集、训练、模型、验证、报告、审计等领域对象 |
| Application | 工作流编排、上下文、命令、状态同步 |
| Services | 数据准备、训练、模型注册、验证、Runtime DATA 导出 |
| Core | 文件 IO、Python 桥、ONNX Runtime、路径、日志 |
| UI | Shell、页面、组件、状态展示，不直接承载业务算法 |
| Python Core | MNE/PyTorch 训练与转换脚本，输出结构化日志 |

代码约束：

- 不再向 `MainComponent` 增加大型业务逻辑。
- 新功能优先进入 Service 或 Page 独立组件。
- UI 只编排视图状态，不直接解析训练数据细节。
- 所有中文 UI 文案继续使用宽字面量或已验证的 UTF-8 安全路径。

## 8. 分阶段开发计划

### P0：当前闭环稳定

目标：两页可用，命令入口干净，训练与推理包生成可烟测。

交付：

- 主导航只保留模型训练、推理模型。
- 命令面板只保留高频业务命令。
- 启动默认进入模型训练。
- Runtime DATA 导出路径、manifest、验证报告可检查。
- Runtime DATA 服务层执行导出前预检、导出后 contract 检查与审计落盘。
- 训练、模型验证、Runtime DATA 导出统一写入 `audit_events.jsonl`，事件结构由 `Core/AuditTrail` 管理。
- 文档建立企业级医疗开发基线。

验收：

- 新用户 1 分钟内能找到“选择数据 -> 开始训练”。
- 无专家模式时看不到采集/预处理/归档命令入口。
- 无有效数据时不会启动训练。

### P1：数据源工程化

目标：训练文件源可治理、可复现。

交付：

- Dataset Manifest 标准化。当前已落地 `Domain::DatasetManifest`、`dataset_manifest.json` 写入、训练前读取与形状一致性阻断。
- NPZ/EDF/BDF/FIF/CSV 导入统一进入标准目录。
- 文件 hash、通道 schema、标签 schema 记录。
- 数据预检结果可保存。

### P2：训练工程化

目标：训练可复现、可恢复、可比较。

交付：

- 训练配置 preset。
- checkpoint 与训练指标持久化。
- ONNX 导出 contract 校验。
- 训练失败后的恢复与诊断。

### P3：推理交付工程化

目标：Runtime DATA 可交付、可验证。

交付：

- Runtime DATA 包结构冻结。
- Golden Sample 回归。
- CPU/DirectML/CUDA EP 兼容性报告。
- 验证报告导出。

### P4：企业治理扩展

目标：支撑团队级、审计级使用。

交付：

- 用户与权限。
- 审计日志查询。
- 项目级版本冻结。
- 插件接口。
- 安装包与发布流程。

## 9. 当前代码改造顺序

1. 收敛入口：导航、命令面板、启动页。
2. 固化 Runtime DATA 包 contract。
3. 给训练和验证增加结构化审计事件。
4. 把 Dataset Manifest 从文档要求落成代码对象。
5. 拆分 `MainComponent` 中仍然保留的历史页面逻辑。
6. 为核心服务补最小单元测试和烟测脚本。

## 10. 本轮开发边界

本轮不做：

- 不重写采集页。
- 不引入登录权限系统。
- 不大规模重构 `MainComponent`。
- 不新增复杂模型库。
- 不承诺医疗认证，只按医疗级工程纪律推进。

本轮只做：

- 建立统一产品与医疗级开发计划。
- 收紧当前主入口和命令入口。
- 保持训练、推理、导出既有功能稳定。
